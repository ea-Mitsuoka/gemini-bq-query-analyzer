# ADR-0003: レポートは顧客 GCS に保存し、期限付き V4 署名付きURLで配信する

- ステータス: Accepted
- 日付: 2026-07-24
- 関連: [design-decisions.md](../design-decisions.md), [ADR-0001](0001-notification-channels.md)

## コンテキスト

生成されるレポートには、顧客の **クエリ本文・スキーマ・コスト情報などの機微情報**が含まれる。これをどこに置き、誰に・どうやって閲覧させるかを決める必要がある。

制約・前提:

- テナントは通常**別組織・別ワークスペース**（クロス組織）。
- 当社（ホスト）が顧客データに恒常アクセスできる状態は、プライバシー・最小権限の観点で避けたい。
- 人間（当社オペレーター／テナント担当者）へバケットの常設 IAM を配ると、退職・異動時の棚卸しや漏洩時の影響範囲が広がる。

## 決定

- レポートは **顧客プロジェクトの GCS バケット**に保存する（データは顧客が保持＝プライバシー優先のデフォルト）。
- 閲覧は **期限付きの V4 署名付きURL**で行い、**人間へバケットの常設 IAM は付与しない**。
  - 署名は Cloud Run SA の **鍵レス signBlob**（SA へ `roles/iam.serviceAccountTokenCreator` を自分自身に付与）。JSON 鍵ファイルは作らない。
  - 有効期限は既定 **7 日**。
- Workflow は成功時に **要点サマリ＋署名付きURL** を返り値として返す。
  - `make run`（当社実行）では実行者の手元に URL が返る（当社への「通知」はこの返り値で満たす。[ADR-0001](0001-notification-channels.md)）。
  - テナント向けは opt-in の Slack 通知に URL を載せる（[ADR-0001](0001-notification-channels.md)）。

## 根拠

- **鍵レス署名**: サービスアカウント鍵ファイルは長期の持ち出し可能なクレデンシャルで、漏洩時の被害が大きい。signBlob（IAM 経由）なら鍵ファイルを一切発行せず、短命トークンで署名でき、鍵管理コストと漏洩面を消せる。
- **署名付きURL**: 常設 IAM を配らずに、必要な人へ・必要な期間だけ・特定オブジェクトへの読み取りを渡せる。期限切れで自動失効するため棚卸し不要。
- **顧客プロジェクトに保存**: データ主権を顧客側に置き、当社は既定でデータを保持しない設計にできる。
- 権限は `objectAdmin` 相当（`objects.list/get/create`）に留め、バケットメタデータ取得（`buckets.get`）に依存しない実装にした（存在確認は `list_blobs(max_results=1)` を使用）。

## 影響 / トレードオフ

- 署名付きURLは**URLを知る者は誰でも**期限内アクセスできる（ベアラー性）。共有経路の管理と、必要十分に短い期限設定が前提になる。既定 7 日はレビュー猶予と失効のバランス。
- `roles/iam.serviceAccountTokenCreator`（自己 signBlob）の付与が必要で、これ自体は強い権限。SA のスコープを分析用途に限定して保つ必要がある。
- URL 失効後は再発行（再実行 or 再署名）が必要。恒常的な閲覧ポータルが欲しくなったら別設計（プロダクト化フェーズ）。

## 検討した代替案

- **人間へバケットの常設 IAM を付与**: クロス組織で恒常アクセスが残り、棚卸し・漏洩時の影響が大きい。→ 不採用。
- **サービスアカウント JSON 鍵で署名**: 長期クレデンシャルの発行・保管・失効が必要で漏洩面が広い。signBlob で代替できる。→ 不採用。
- **公開バケット / 認証なし URL**: 機微情報を公開面に置くことになり論外。→ 不採用。
- **バケットメタデータ取得（`get_bucket`）に依存した存在確認**: `buckets.get` を要求し必要権限が広がる。`list_blobs` で代替。→ 不採用。
