# 設計方針・決定事項

通知と提供モデルに関する決定事項をまとめる（実装の背景・意図の記録）。

個別の重要決定は [ADR](adr/) を参照:

- [ADR-0001: 通知チャネルの使い分け](adr/0001-notification-channels.md)

## 実行契機

| 契機                     | 用途                                                 | 受け手                         |
| :----------------------- | :--------------------------------------------------- | :----------------------------- |
| **Cloud Scheduler**      | テナント向けの定期レポート                           | テナント（周期配信）           |
| **`make run TENANT=..`** | オンデマンド（依頼時・修正後の再監査・当社コンサル） | **実行者（当社オペレーター）** |

- 両者は独立して共存する。
- `make run` は**当社（オペレーター）用の CLI** であり、顧客には渡さない。顧客セルフサービスのオンデマンド実行を提供する場合は、CLI ではなく **テナント限定認可＋レート/コスト制限を備えた顧客向けトリガ（ポータル/認証付きAPI）** を別途用意し、同じ Workflow を再利用する（プロダクト化フェーズ）。

## 成功時の結果配信

- レポートは**顧客プロジェクトの GCS バケット**に保存する（データは顧客が保持＝プライバシー優先のデフォルト）。
- 閲覧は**期限付きの V4 署名付きURL**で行う。人間へバケットの常設 IAM は付与しない（最小権限・クロス組織の恒常アクセスを作らない）。
  - 署名は Cloud Run SA の鍵レス signBlob（SA へ `roles/iam.serviceAccountTokenCreator` を付与）。有効期限は既定7日。
- **Workflow は成功時に結果（要点サマリ＋署名付きURL）を返り値として返す**。
  - `make run`（当社実行）では、**実行した当社オペレーターの手元に署名付きURLが返る**（＝当社への「通知」はこの返り値で満たす。別チャネルへの自動 push はしない）。
  - Scheduler 実行では返り値は使われない。
- **テナント向け通知**は、`slack_webhook_secret_name` が設定されたテナントにのみ Slack 通知（opt-in、要点＋署名URL）。空なら送らない。メール配信は将来の拡張（メール送信基盤が必要）。

## 当社（ホスト）への結果共有

- **自動連動させない**。当社共有は「実行者＝当社」のときに返り値で得るか、明示同意（`share_with_host`）に基づく。
- テナントがセルフ実行する将来モデルでは、既定はテナントのみ。当社共有は per-run 明示（`SHARE_HOST`）× standing consent（`share_with_host`）の二段ゲート。
- 背景: レポートには顧客のクエリ本文・スキーマ等の機微情報が含まれるため、当社アクセスは明示同意＋最小権限（署名URL）とする。

## 失敗通知（社内運用向け）

- 失敗検知は **Workflow 層**で行う（Job が exit 0 でも Workflow が失敗するケースがあるため）。
  - Workflow は `try/except` で、失敗時に構造化ログ `ANALYZER_FAILURE tenant=... error=...` を出して re-raise する。
  - Cloud Run Job も、復旧不能なエラー（設定不備・レポートバケットアクセス不可・SQL 読込失敗）では **exit 1** にしてサイレント失敗を防ぐ。
- 通知は **Cloud Monitoring のログベースアラート → Email 通知チャネル → Slack チャンネルの Integration メールアドレス**（当社ワークスペース）。
  - Email チャネルなので Slack OAuth アプリ（情シス承認）は不要。
  - 宛先は `base_config.ini` の `alert_notification_email`。空なら通知リソースを作らない。

## 監視の方針

- **変化検知/閾値監視は作らない**（構築・運用コストが肥大するため）。定期はテナントへの定期レポートに留め、**コンサルは依頼ベース（pull）**。
